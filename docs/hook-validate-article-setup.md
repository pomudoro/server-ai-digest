# Stop-хук `validate-article.sh`: итоги настройки

Короткая сводка того, что удалось настроить и проверить в `.claude/hooks/validate-article.sh` и связанной регистрации в `settings.json`.

## Регистрация хука

В `.claude/settings.json` добавлен блок:

```json
"Stop": [
  {
    "hooks": [
      {
        "type": "command",
        "command": "bash \"$CLAUDE_PROJECT_DIR/.claude/hooks/validate-article.sh\""
      }
    ]
  }
]
```

Особенности выбранной формы для Windows 11 / git-bash:

- Явный вызов `bash` (а не путь к `.sh` напрямую) — на Windows нет shebang-диспетчера.
- Переменная `$CLAUDE_PROJECT_DIR` — переносимо между машинами, без хардкода абсолютного пути.
- Кавычки вокруг пути — страховка от пробелов.

## Правки самого скрипта

| Правка          | Что было                                                                | Что стало                                                                                                   |
| --------------- | ----------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| Путь к контенту | `find content/ ...` (папка в проекте отсутствует, хук всегда пропускал) | `find src/content/blog/ ...`                                                                                |
| Поле обложки    | `^(cover\|image):`                                                      | `^(cover\|image\|heroImage):` — согласовано со схемой `src/content.config.ts`, где используется `heroImage` |

## Проверенные ветки валидации

Ручная симуляция через `echo '{...}' \| bash .claude/hooks/validate-article.sh`:

| #   | Сценарий                        | Ожидание | Результат |
| --- | ------------------------------- | -------- | --------- |
| 1   | Нет `title:`                    | exit 2   | OK        |
| 2   | Нет `description:`              | exit 2   | OK        |
| 3   | Есть `heroImage:` (новая ветка) | exit 0   | OK        |
| 4   | Есть все четыре поля            | exit 0   | OK        |

Живой тест через `claude --debug` + битый `_hook-test.md` подтвердил: хук корректно блокирует Stop с `exit 2` и отправляет stderr пользователю.

## Известные слабости (оставлены осознанно)

1. **Ложноположительный `source`.** Любой `https?://` в теле markdown засчитывается за источник — `grep` не различает «URL в поле `source`» и «URL в обычной ссылке».
2. **Пустые значения ключей.** `title:` без текста проходит проверку: регулярка `^title:` требует только наличие ключа, а не значения.

Оба случая принято оставить как есть: в текущем workflow статьи пишутся руками с заполненными полями, а полноценная YAML-валидация потребовала бы `yq` или Python-парсера.

## Про ветку `stop_hook_active`

Guard работает, но только в non-interactive режиме (`claude -p`, Agent SDK, CI) — там после блокировки модель авто-продолжается и на повторном Stop прилетает `stop_hook_active: true`. В интерактивном TTY этой фазы нет: stderr идёт пользователю, повторного Stop не случается. Поэтому тестировать guard нужно через `claude -p`, а не в живом чате.

Подробный разбор с логами из обоих режимов — в [`stop-hook-active-issue.md`](./stop-hook-active-issue.md).
