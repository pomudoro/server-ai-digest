# Тарифы Claude и как считаются токены

Памятка про то, как Anthropic тарифицирует запросы к Claude API: что такое input, output, cache read и cache write, чем отличается 5-min кэш от 1-hour, и как это работает в Claude Code.

## Тарифы (за 1M токенов)

| Тип                | Opus 4.x              | Sonnet 4.x | Haiku 4.5 |
| ------------------ | --------------------- | ---------- | --------- |
| Input              | $15                   | $3         | $1        |
| Output             | $75                   | $15        | $5        |
| Cache write 5-min  | $18.75 (≈1.25× input) | $3.75      | $1.25     |
| Cache write 1-hour | $30 (≈2× input)       | $6         | $2        |
| Cache read         | $1.50                 | $0.30      | $0.10     |

Опорные пропорции, которые удобно держать в голове:

- Cache write 5-min ≈ **1.25× input**
- Cache write 1-hour ≈ **2× input**
- Cache read ≈ **0.1× input** (в 10 раз дешевле обычного input)
- Output всегда **5× input**

## Чем отличаются input, output, cache read/write

|                 | Что это                                                                                            | Направление                         | Когда возникает                            |
| --------------- | -------------------------------------------------------------------------------------------------- | ----------------------------------- | ------------------------------------------ |
| **Input**       | Текст, который ты отправил модели (промпт, история, файлы)                                         | user → API                          | Каждый запрос                              |
| **Output**      | Текст, который сгенерировала модель в ответе                                                       | API → user                          | Каждый запрос                              |
| **Cache write** | Часть **input**, помеченная флагом `cache_control` — API сохраняет её как переиспользуемый префикс | user → API (с пометкой «закешируй») | Только при первом запросе с этим префиксом |
| **Cache read**  | Тот же префикс, прочитанный из кэша на следующих запросах                                          | переиспользование                   | На последующих запросах в пределах TTL     |

**Output никогда не превращается в cache write.** Это разные направления: output — что модель сказала, cache write — кусок твоего input, помеченный «сохрани этот префикс».

Один и тот же токен не может быть одновременно input и cache write — выбор определяется наличием `cache_control` в запросе.

## 5-min vs 1-hour cache

Это **TTL префикса** — сколько Anthropic держит его в кэше:

- **5-min cache**: префикс живёт 5 минут после последнего обращения. Каждое чтение продлевает TTL. Запись дешевле (1.25× input).
- **1-hour cache**: префикс живёт 1 час. Запись дороже (2× input), но если запросы редкие — выгоднее.

Выбор делает клиент через параметр `cache_control: { type: "ephemeral", ttl: "5m" | "1h" }`. Не зависит от ситуации.

- Интерактивные клиенты (Claude Code и т.п.) → **5-min**. Запросы идут плотно, кэш не протухает.
- Редкие запросы (агент раз в 20 минут) → **1-hour** имеет смысл, иначе 5-min каждый раз протухает.

## Как это выглядит за один диалог (новая сессия, Opus)

### Ход 1 — первый вопрос пользователю

```
[system prompt]    ← cache_control
[tools schema]     ← cache_control
─── граница кэша ───
[твой первый вопрос]
```

| Что            | Тариф                                              |
| -------------- | -------------------------------------------------- |
| system + tools | cache write 5-min ($18.75/M) — кэша ещё нет, пишем |
| первый вопрос  | **обычный input ($15/M)**, НЕ cache write          |
| ответ модели   | output ($75/M)                                     |

### Ход 2 — уточняющий вопрос

```
[system prompt]              ← cache_control (был в кэше)
[tools schema]               ← cache_control (был в кэше)
─── префикс из хода 1 ───
[твой первый вопрос]
[ответ модели на ход 1]
                             ← новая cache_control
─── новая граница ───
[твой уточняющий вопрос]
```

| Что                          | Тариф                                              |
| ---------------------------- | -------------------------------------------------- |
| system + tools               | cache read ($1.50/M) — уже в кэше                  |
| первый вопрос + первый ответ | cache write 5-min ($18.75/M) — пристёгиваем к кэшу |
| уточняющий вопрос            | **обычный input ($15/M)**, НЕ cache write          |
| новый ответ                  | output ($75/M)                                     |

**Вывод:** свежее сообщение пользователя всегда идёт как обычный input. Cache write — это про префикс, который имеет смысл переиспользовать на следующем запросе. Самый хвост никогда не кэшируется на текущем ходе.

На ходе 3 этот уточняющий вопрос (вместе с ответом) превратится в cache write, на ходе 4 — в cache read.

## Шпаргалка: куда пойдёт каждый токен

```
   Откуда токен берётся?
            │
            ▼
   ┌────────────────┐
   │ Это вход модели│ ──► input-семейство
   │ (то что я шлю) │
   └────────────────┘
            │
            ▼
   Помечен cache_control в запросе?
            │
       ┌────┴────┐
       │ нет     │ да
       ▼         ▼
   обычный    есть в кэше?
   input          │
              ┌───┴───┐
              │ нет   │ да
              ▼       ▼
          cache    cache
          write    read

   ┌─────────────────────┐
   │ Что сгенерила модель│ ──► output (всегда дороже всего)
   └─────────────────────┘
```

## Применительно к Claude Code

- В начале сессии видна большая волна **cache write** (~100–200k токенов) — это инициализация system prompt + tools.
- Каждый ход добавляет немного cache write (растущая история) и читает много cache read.
- **Output** — то, что Claude печатает в чат: код, объяснения, tool calls.
- Subagent'ы (writer, news-scout, cover-artist, page-builder) каждый раз инициализируют свой контекст → большой объём cache write на Sonnet. Это часто делает Sonnet сравнимым по стоимости с Opus в digest-сессии.
- Claude Code использует 5-min TTL по умолчанию.

## Где смотреть свои расходы

В Claude Code: вкладка **Usage** в экране статуса сессии. Видно:

- Total cost, API duration, wall duration
- Раскладка по моделям: input / output / cache read / cache write → стоимость
- Доля subagent-heavy сессий и >150k context
